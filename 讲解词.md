# 北信科旧物交换平台 — 15分钟讲解词

> 4人分工，每人约3-4分钟

---

## 👤 第1人：项目概述 & 数据库设计（3分钟）

大家好，我们做的是一个**北信科校园旧物交换平台**。

**项目背景**：校园里很多闲置教材、生活用品，毕业季扔掉很浪费。我们想做一个只有校内同学能用的二手交易平台，安全、简单。

**技术栈**：前端是原生 HTML/CSS/JS，后端用 Node.js + Express，数据库用 MySQL，身份认证用 JWT + bcrypt。

**数据库设计**：核心就两张表。第一张 `users` 表存储用户信息，密码用 bcrypt 哈希存储——数据库里存的是 `$2a$10$...` 这种不可逆的密文。第二张 `goods` 表存储商品，有名称、价格、成色、交易地点这些字段，用外键关联到 users 表，seller_id 是卖家、buyer_id 是买家。后来我们还加了第三张 `user_sessions` 表来做设备登录限制。初始化脚本里预置了8个用户和6条示例商品数据。

**分工**：我们四个人，有人负责数据库和后端接口，有人负责前端页面改造，有人负责 GitHub 仓库协作，还有人负责网络部署和文档。

好，技术细节交给后面的同学展开。

## 👤 第2人：后端架构 & API & 认证（4分钟）

我来详细讲后端。

**整体架构**：Express 作为 Web 框架，entry 文件 server.js 启动后监听 3000 端口。代码按职责分成四个文件夹——config 放数据库连接，middleware 放中间件，routes 放接口逻辑，utils 放工具函数。

**连接池**：config/db.js 用了 mysql2 的 createPool，最大10个连接。为什么用连接池不用单连接？因为 Node 是异步的，多个请求同时进来，单连接会排队阻塞，连接池可以复用。

**三层中间件链**：请求进来，先过 `express.json()` 解析 JSON，然后过 `auth` 中间件验证 JWT，如果是管理员接口再过 `adminOnly` 检查角色。任何一层不通过就直接返回 401 或 403。

**身份认证**：用的是 JWT + bcrypt。用户登录时，bcrypt 把输入的密码和数据库的哈希做比对——注意密码从不出现在代码里，也不在 JWT 中传输。验证通过后签发 JWT，里面存了用户 ID、用户名、角色，还有一个 jti 唯一标识。24小时后过期。

**设备限制**：我们后来加了一个功能——每个用户最多同时2台设备登录。实现原理是在 user_sessions 表里记录每个 JWT 的 jti。登录时先清理过期会话，再统计当前活跃数，超过2个就删掉最老的那条，把旧设备踢下线。登出时直接删对应记录，下次请求中间件发现 jti 不存在就返回 401。这样登出和踢下线都是即时生效的。

## 👤 第3人：前端设计 & 安全防护（4分钟）

我来讲前端和安全性。

**前端方案**：我们没有用 Vue 或 React，而是保留了原始的单文件 HTML。这是有意为之——不用构建工具，clone 下来直接能跑，降低参与门槛。页面是个 SPA，四个 Tab：我要买、我要卖、管理后台、我的信息。

**前后端对接**：原来数据全存在浏览器 localStorage 里，密码明文写在 JS 中——这非常不安全。我们把它全部改成 API 调用。封装了一个 `api()` 函数，自动在每个请求的 Header 里带上 `Authorization: Bearer <token>`。所有数据都从后端按需拉取，localStorage 只存一个 token。

**五层安全防护**：这是我觉得最值得说的。

第一层，**SQL 注入防护**：所有数据库查询都用 `pool.execute(sql, [params])` 这种参数化查询，用 `?` 占位符。用户输入永远不会被拼接到 SQL 语句里，即使输入 `admin' OR '1'='1` 也只会被当作普通字符串。

第二层，**密码安全**：bcrypt 哈希加随机 salt，cost 设为10，每次验证大约50毫秒。数据库泄露也无法反推密码。

第三层，**JWT 防篡改**：token 的第三部分是服务器用密钥算出来的签名。任何人改了 payload 里的内容，签名就对不上了，直接拒绝。

第四层，**角色权限**：前端有一个选择管理员还是普通用户的下拉框，但那个只是参考。真正决定角色的是数据库查出来的值。JWT 里写的也来自数据库。管理员接口还加了 adminOnly 中间件，双重保险。

第五层，**XSS 防护**：所有 API 返回数据之前都经过 sanitizeObject 处理，`<script>` 会变成 `&lt;script&gt;`，浏览器只显示不执行。

## 👤 第4人：GitHub 协作 & 网络部署 & 问题解决（4分钟）

我来讲三个部分：GitHub 工作流、实际遇到的问题、以及如何解决的。

**GitHub 分支策略**：我们的仓库在 Bistu-OSSDT-2026 组织下。main 分支是稳定版本，所有新功能都从 main 开 feature 分支开发。命名规范是 `feature/功能名`，比如 `feature/device-limit` 是设备限制、`feature/ngrok-tunnel` 是内网穿透。开发完提 Pull Request，review 通过后合并到 main。

**真实踩过的坑**：我们遇到一个很典型的多分支协作冲突。仓库里有个 `code-debugger` 分支也在改 auth.js，main 分支又被我们合并了设备限制功能。两边改了同一个文件的同一个位置——登录逻辑和返回体。Git 自动合并失败，auth.js 里出现了 `<<<<<<<` 冲突标记，导致代码里有重复的 user 字段，服务直接启动报语法错误。解决办法是——全部保留 main 的最新版本，因为 main 包含了更完整的功能，然后在 main 上修掉残留代码重新提交。

**校园网问题**：另一个大问题是局域网访问。我们配置了服务监听 0.0.0.0，加了 Windows 防火墙规则放行 3000 端口，本机自测完全正常。但同学就是访问不了 `http://10.153.xxx.xxx:3000`。排查后发现是校园网开了 AP 隔离——同一 WiFi 下设备之间不允许直接通信，这是学校网络设备层面的限制，我们改不了。

**解决方案**：内网穿透。我们试了 localtunnel，一行 `npx lt --port 3000` 就能生成公网链接，但免费版不太稳定，会断。最后用了 ngrok——注册免费账号，拿到 authtoken，运行 `ngrok http 3000`，得到一个固定的 HTTPS 地址。这相当于在公网 ngrok 服务器和我们本地之间打了个加密隧道。好处是任何人任何网络都能访问，HTTPS 加密比原来局域网 HTTP 明文更安全。唯一的代价是多了一跳中转，延迟多了几十毫秒，但浏览网页完全感觉不到。

**总结**：从单机 demo 到可以多人协作的项目，我们完整走了一遍——数据库设计、后端接口、安全防护、Git 分支管理、冲突解决、网络部署。虽然中间踩了不少坑，但每个坑都是一次学习。谢谢大家。
