
### 2.3 依赖包详解 (`package.json`)

| 包名 | 版本 | 用途 | 关键 API |
|------|------|------|----------|
| **express** | ^4.21.2 | Web 框架 | `express()`, `Router()`, `app.use()` |
| **mysql2** | ^3.12.0 | MySQL 驱动 | `createPool()`, `pool.execute()` (Promise) |
| **jsonwebtoken** | ^9.0.2 | JWT 令牌 | `jwt.sign()`, `jwt.verify()` |
| **bcryptjs** | ^2.4.3 | 密码哈希 | `bcrypt.hashSync()`, `bcrypt.compareSync()` |
| **dotenv** | ^16.4.7 | 环境变量 | `dotenv.config()` → `process.env.XXX` |

**为什么选 `mysql2` 而不是 `mysql`？**
- `mysql` 是回调风格，`mysql2` 原生支持 Promise
- `mysql2` 的 `execute()` 方法内置参数化查询（防注入）
- `mysql2` 支持 `utf8mb4` 编码，正确处理中文和 emoji

**为什么选 `bcryptjs` 而不是 `bcrypt`？**
- `bcryptjs` 纯 JS 实现，无需编译原生模块
- 跨平台零安装问题（Windows/Linux/macOS 通用）
- 性能对学习项目完全够用

### 3.3 数据库设计详解

#### `users` 表

```sql
CREATE TABLE users (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  username    VARCHAR(50)  NOT NULL UNIQUE,     -- 登录名，唯一约束
  password    VARCHAR(255) NOT NULL,            -- bcrypt 哈希固定 60 字符，预留空间
  role        ENUM('admin', 'user') NOT NULL,   -- 枚举约束，防止非法值
  school      VARCHAR(100) DEFAULT '',          -- 预留给学校信息
  created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**设计要点：**
- `UNIQUE` 约束确保用户名不重复
- `ENUM` 类型让数据库层面就阻止了非法角色值
- `password` 用 `VARCHAR(255)` 而非 `CHAR(60)`——bcrypt 哈希长度可能随 cost 变化

#### `goods` 表

```sql
CREATE TABLE goods (
  id              INT AUTO_INCREMENT PRIMARY KEY,
  name            VARCHAR(200)  NOT NULL,
  price           DECIMAL(10,2) NOT NULL,   -- 精确小数，避免浮点误差
  `condition`     VARCHAR(20) NOT NULL,     -- 反引号：condition 是 MySQL 保留字
  location        VARCHAR(50)  NOT NULL,
  description     TEXT,                     -- 长文本
  image_url       VARCHAR(500) DEFAULT '',
  seller_id       INT NOT NULL,
  buyer_id        INT DEFAULT NULL,         -- 可空（商品未售出时）
  status          ENUM('onsale','offline','sold') NOT NULL DEFAULT 'onsale',
  offline_reason  TEXT DEFAULT NULL,
  notified        TINYINT(1) DEFAULT 0,     -- 布尔值：0=未通知 1=已通知
  created_at      DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at      DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

  FOREIGN KEY (seller_id) REFERENCES users(id),  -- 级联保护
  FOREIGN KEY (buyer_id)  REFERENCES users(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**设计要点：**
- `DECIMAL(10,2)` 而非 `FLOAT`：金额运算不丢精度
- `` `condition` `` 加反引号是因为这是 MySQL 保留关键字
- `buyer_id` 可空，在售商品没有买家
- `notified TINYINT(1)` — MySQL 没有真正的 BOOLEAN 类型
- `ON UPDATE CURRENT_TIMESTAMP` — 任何字段更新时自动更新时间戳

### 4.2 bcrypt 密码哈希机制

```javascript
// 哈希生成 (init.sql 创建时)
const hash = bcrypt.hashSync('123', 10);
// → "$2a$10$Z1478//uA75d4qDLAzA5TOEERyMoFQ4KAhGvi2Zm3u4NLyqLVkFJ2"
//     └─┬─┘└─┬─┘└─────────────────┬─────────────────────────┘
//      版本  cost      22字符 salt      31字符 hash
//      (2a)  (10)

// 密码验证 (每次登录)
const match = bcrypt.compareSync('123', storedHash);
// 1. 从 storedHash 中提取 salt（前 29 字符）
// 2. 用同样的 salt + cost 重新哈希 '123'
// 3. 比较两个哈希是否一致
```

**cost=10 的含义：** 算法内部迭代 2^10 = 1024 次。每次验证约 50-100ms，既能有效防暴力破解，又不影响用户体验。

### 4.3 JWT Token 结构分析

```javascript
// 签发 (server side)
jwt.sign(
  { id: 2, username: '张三', role: 'user' },  // Payload - 公开信息
  'bistu-exchange-secret-key-2024',            // Secret - 仅服务端知晓
  { expiresIn: '24h' }                         // 过期时间
);

// 生成的 Token（三部分，base64 编码，点号分隔）
// eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9          ← Header
// .
// eyJpZCI6MiwidXNlcm5hbWUiOiLlvKDkuIkiLCJyb2xlIjoidXNlciIsImlhdCI6MTc4MzA4MzA4NCwiZXhwIjoxNzgzMTY5NDg0fQ  ← Payload
// .
// qVUeuwm0j8xUM1nYwc_-4OpypnWH8drBablCVZGfL3w    ← Signature（防篡改关键）
```

**验证时**：`jwt.verify(token, secret)` 会用 secret 重新计算 Signature，如果和 Token 中的一致 → 通过；如果不一致（哪怕 Payload 改了一个字符）→ 拒绝。

## 5. 安全防护体系

### 5.1 五层防御模型

```
        ┌─────────────────────────────────┐
  第5层 │  输出层：HTML 转义 (sanitize.js) │  ← XSS 防护
        ├─────────────────────────────────┤
  第4层 │  权限层：adminOnly 中间件         │  ← 权限隔离
        ├─────────────────────────────────┤
  第3层 │  认证层：JWT 签名验证             │  ← 身份验证
        ├─────────────────────────────────┤
  第2层 │  存储层：bcrypt 密码哈希          │  ← 数据泄露防护
        ├─────────────────────────────────┤
  第1层 │  传输层：参数化查询 (mysql2)      │  ← SQL 注入防护
        └─────────────────────────────────┘
```

### 5.2 SQL 注入防护

**原理**：参数化查询将 SQL 结构与数据分离，驱动层保证数据不会被当作 SQL 执行。

```javascript
// ❌ 危险：字符串拼接（本项目中没有这种写法！）
const sql = `SELECT * FROM users WHERE username = '${input}'`;
// 输入 admin' OR '1'='1 → SELECT * FROM users WHERE username = 'admin' OR '1'='1'

// ✅ 安全：参数化查询（项目中全部使用）
const sql = 'SELECT * FROM users WHERE username = ?';
const [rows] = await pool.execute(sql, [input]);
// 输入 admin' OR '1'='1 → 当作普通字符串处理，查询 username = "admin' OR '1'='1"
```

### 5.3 XSS 防护

**原理**：将用户输入的 HTML 特殊字符转为实体编码，浏览器显示原样但不执行。

```javascript
// utils/sanitize.js
function escapeHtml(str) {
  return str
    .replace(/&/g, '&amp;')    // & → &amp;
    .replace(/</g, '&lt;')     // < → &lt;
    .replace(/>/g, '&gt;')     // > → &gt;
    .replace(/"/g, '&quot;')   // " → &quot;
    .replace(/'/g, '&#39;');   // ' → &#39;
}
// 输入: <script>alert('XSS')</script>
// 输出: &lt;script&gt;alert(&#39;XSS&#39;)&lt;/script&gt;
// 浏览器显示为文本，不执行为脚本
```

**应用范围**：所有 API 返回的数据都经过 `sanitizeObject()` 处理——用户输入的商品名、描述、地址等字段在返回给前端前已全部转义。

### 5.4 角色伪造防护

```javascript
// 关键安全决策：角色完全由服务端数据库决定
// routes/auth.js 登录时
const [rows] = await pool.execute(
  'SELECT id, username, password, role FROM users WHERE username = ?',
  [username]
);
const user = rows[0];

// JWT 中写入的是数据库查到的 role，不是前端的 loginRole
const token = jwt.sign(
  { id: user.id, username: user.username, role: user.role },  // ← 来自数据库
  JWT_SECRET
);
```

**验证点**：即使用户用浏览器 DevTools 把 `loginRole` 改成 `admin`，JWT 里的 `role` 仍将是数据库里的 `"user"`。

### 6.2 RESTful 设计原则具体体现

| 原则 | 具体做法 |
|------|----------|
| **统一资源标识** | `/api/goods/:id` 而非 `/api/getGoods?id=xxx` |
| **HTTP 方法语义** | GET=查询, POST=创建, PUT=更新 |
| **状态码语义** | 200=成功, 201=创建成功, 400=参数错误, 401=未登录, 403=无权限, 404=不存在 |
| **JSON 响应格式** | 统一 `{error: "..."}` 错误格式 |
| **无状态** | 每次请求独立验证 JWT，服务器不存 session |

## 7. 前端技术分析

### 7.1 技术选型

| 方面 | 实现方式 | 原因 |
|------|----------|------|
| 页面渲染 | 原生 DOM 操作 | 单文件 HTML，零构建 |
| 网络请求 | `fetch()` API | 原生异步，无需引入 axios |
| Token 存储 | `localStorage` | 跨标签页持久化 |
| 状态管理 | 全局变量 `token`, `currentUser` | 简单场景够用 |
| CSS 布局 | CSS Grid + Flexbox | 现代化，无框架依赖 |
| 模板 | ES6 模板字符串 `` `${}` `` | 代替 `innerHTML` 拼接 |

### 7.2 前端安全改造

| 原版问题 | 改后方案 |
|----------|----------|
| 密码明文在 JS 中 (`users` 数组) | 删除 → API 验证 |
| 商品数据存 `localStorage` | 删除 → API 按需拉取 |
| 登录全靠前端判断 `found.role` | JWT 解密 `currentUser.role` |
| `innerHTML` 直接拼接用户输入 | 服务端 `sanitizeObject()` 后返回 |

```
/index.html
│
├── loginPage           登录页（未登录时显示）
```
